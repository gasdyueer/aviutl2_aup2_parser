# AUP2 工程文件格式说明

基于三个真实 AUP2 项目文件（test / simple / complex）的解析结果分析。

## 1. 文件格式概述

AUP2 是 AviUtl 的项目文件格式，纯文本，使用类似 INI 的 `[区块]` + `属性=值` 结构。编码通常为 UTF-8（含日文字符），换行符为 `\r\n`（Windows 原生格式）。

### 基础语法

```
[section]            ← 区块头
key1=value1          ← 键值对
key2=value2
```

- 区块之间没有显式的嵌套标记，嵌套关系由**区块类型**和**出现顺序**隐含决定
- 空行被忽略
- `#` 注释在 AUP2 格式中未观察到

## 2. 逻辑层次结构

```
[project]                    ← 项目元信息（1 个）
  └── [scene.N]              ← 场景（1 ~ N 个）
        ├── 场景属性
        ├── [layer.M]        ← 图层分组/文件夹（可选，0 ~ N 个）
        │     └── [Y]        ← 普通对象
        └── [Y]              ← 普通对象（0 ~ N 个）
              ├── 对象属性 (layer, frame, …)
              └── [Y.Z]      ← 效果/滤镜（0 ~ N 个）
                    └── 效果属性
```

**关键规则**：`[scene.X]` 之后出现的对象和效果，在遇到下一个 `[scene.Y]` 或文件结尾之前，都属于该场景。

## 3. 区块类型

### 3.1 `[project]` — 项目区块

文件第一个区块。包含项目级元信息。

| 属性 | 类型 | 说明 |
|------|------|------|
| `file` | string | 项目文件的原始绝对路径 |
| `display.scene` | int | 当前显示的场景 ID |

### 3.2 `[scene.N]` — 场景区块

`N` 是场景序号（从 0 开始），场景定义在文件中的出现顺序与其序号一致，但**场景内部 ID (`scene=`) 可能与序号不同**。

| 属性 | 类型 | 说明 |
|------|------|------|
| `scene` | int | 场景内部 ID（可与区块序号不同） |
| `name` | string | 场景名称 |
| `video.width` | int | 视频宽度（像素） |
| `video.height` | int | 视频高度（像素） |
| `video.rate` | int | 帧率（fps） |
| `video.scale` | int | 缩放比例（100 = 100%） |
| `audio.rate` | int | 音频采样率（Hz） |
| `cursor.frame` | int | 光标所在帧 |
| `display.frame` | int | 显示起始帧 |
| `display.layer` | int | 显示起始图层 |
| `display.zoom` | int | 显示缩放（100000 = 100%） |
| `display.order` | int | 显示排序 |
| `display.camera` | string | 摄像机设置（通常为空） |

**实际观察**：`complex.aup2` 中 scene.2 的内部 ID 为 3，scene.3 的内部 ID 为 2 —— 区块序号和内部 ID 可以不一致，AviUtl 使用内部 ID 进行场景关联。

### 3.3 `[Y]` — 对象区块

`Y` 是对象 ID（整数，从 0 开始全局递增）。不同场景中的对象共享同一个 ID 序列。

| 属性 | 类型 | 说明 |
|------|------|------|
| `layer` | int | 图层编号（0 = 最底层，越大越靠上） |
| `frame` | int list | 关键帧位置，见 [帧/关键帧系统](#7-帧关键帧系统) |
| `scene` | int | 所属场景 ID（可选，未显式声明时由出现位置推断） |
| `group` | int | 所属图层分组 ID（可选） |
| `focus` | int | 是否被选中（可选） |
| `camera` | int | 摄像机控制（可选） |

### 3.4 `[Y.Z]` — 效果区块

`Y` 是父对象 ID，`Z` 是效果序号（从 0 开始，每个对象独立计数）。效果按 `[Y.0]`, `[Y.1]`, … 的顺序应用到对象上，**顺序决定渲染管线中的叠加次序**。

### 3.5 `[layer.M]` — 图层分组区块

AviUtl 的图层文件夹功能。将多个对象组织为一个分组。

| 属性 | 类型 | 说明 |
|------|------|------|
| `layer` | int | 图层编号 |
| `scene` | int | 所属场景 ID |
| `hide` | int | 是否隐藏（0/1） |

图层分组内可以包含普通对象 `[Y]`。

## 4. 数据类型

### 4.1 整数

```
1920
0
48000
```

### 4.2 浮点数

```
100.00      → 浮点数 100.0
0.00        → 浮点数 0.0
-429.70     → 浮点数 -429.7
```

注意：AUP2 **保留原始精度**（如 `100.00` vs `100.0`），数值的小数位数可能有语义含义或纯粹是编辑器的格式化习惯。

### 4.3 逗号分隔的数值列表

```
0,411                           → [0, 411]       (纯整数列表)
0.000,13.723                    → [0.0, 13.723]  (纯浮点数列表)
0,2,16,26,42,46                 → [0, 2, 16, 26, 42, 46]  (较长的帧列表)
```

用于 `frame` 属性和某些效果参数。

### 4.4 关键帧字符串（混合类型）

```
58.000,95.000,直線移動(時間制御),0|0,0,0,1
0.000,4.150,再生範囲,0
-710.0,490.0,直線移動(時間制御),0|0,0.511905,1,0
```

这是一种**自定义编码格式**，以逗号分隔，字段含义：

| 位置 | 内容 | 示例 |
|------|------|------|
| 1 | 起始值 | `58.000` |
| 2 | 结束值 | `95.000` |
| 3 | 插值方式 | `直線移動(時間制御)`, `再生範囲`, `加速`, `減速` 等 |
| 4 | 关键帧数据 | `0\|0,0,0,1`（管道符分隔的多段参数） |

关键帧字符串**不是纯数值列表**，不能简单解析为 `[float, float, ...]`。它们出现在所有支持动画的效果属性中（位置、旋转、缩放、透明度等）。

### 4.5 字符串

```
e0e1dd              ← 颜色值（十六进制）
通常                ← 合成模式
背景                ← 图形类型
```

### 4.6 空值

```
display.camera=     ← 空字符串
```

## 5. 效果系统

### 5.1 效果链机制

每个对象可以有 0 到多个效果，构成一条**效果链**（effects chain）：

```
[Y]                  ← 对象
  [Y.0]              ← 效果 0（先应用）
  [Y.1]              ← 效果 1
  [Y.2]              ← 效果 2（后应用）
  ...
```

同类型效果可以重复叠加（如一个对象上有 5 个连续的 `グラデーション` 效果）。

### 5.2 效果对象 vs 效果属性

每个效果由**两层**构成：容器和内容。

```
"effect.0": {                     ← 效果对象（容器）
    "effect.name": "図形",        ← 效果属性（内容，只不过恰好叫 effect.name）
    "サイズ": 100,                ← 效果属性
    "色": "e0e1dd"                ← 效果属性
}
```

**效果对象**（Effect Object）即 `[Y.Z]` 区块本身——`effect.0`、`effect.1`。它是一组属性的容器：

- 在文件中的位置决定了它在效果链中的次序（渲染顺序）
- 通过区块命名 `[Y.Z]` 关联到父对象 `object.Y`
- `Z` 是同一对象内的效果序号（从 0 递增）

**效果属性**（Effect Property）即效果对象内部的键值对。`effect.name` 只是其中之一，没有任何格式层面的特殊待遇——它和 `サイズ`、`色` 一样，是一个普通的 key=value 属性。它的值恰好告诉 AviUtl 用哪个效果插件来渲染，但解析器不需要特殊对待它。

在文件中表现为：

```
[2]                    ← 对象 object.2
layer=1
frame=97,146
[2.0]                  ← 效果对象（容器开始）
effect.name=図形         ← 效果属性
図形の種類=背景           ← 效果属性
サイズ=100               ← 效果属性
[2.1]                  ← 下一个效果对象
effect.name=標準描画      ← 效果属性
透明度=0.00              ← 效果属性
```

这与场景和对象的关系是同一套嵌套模式的递归：

```
scene.0 : object.2   =   object.2 : effect.0
场景属性 : 对象属性   =   对象属性 : 效果属性
```

## 6. 图层系统

### 6.1 图层编号

- `layer=0` 是最底层（背景）
- 编号越大越靠前（覆盖下层）
- 图层可以不连续（simple 项目中 layer 范围 0~45，中间大量空白）
- 不同场景可以复用相同的图层号

### 6.2 图层分组 `[layer.M]`

将多个连续对象组织为分组，分组本身有 `hide` 属性控制可见性。分组内的对象通过 `group` 属性关联到分组。分组可以嵌套在场景内的任意位置。

## 7. 帧/关键帧系统

### 7.1 `frame` 属性

对象的 `frame` 属性是一个**整数列表**，含义取决于列表长度：

| 长度 | 含义 | 示例 |
|------|------|------|
| 2 | `[startFrame, endFrame]` — 对象的显示区间 | `[0, 411]` |
| 3 | `[startFrame, keyFrame1, endFrame]` — 含一个中间关键帧 | `[82, 91, 111]` |
| 6 | 含多个中间关键帧 | `[0, 2, 16, 26, 42, 46]` |
| 7 | 同上 | 含更复杂的时间线 |

起始帧为第一帧，结束帧为最后一帧，中间帧为关键帧分割点。对象只在 `startFrame` 到 `endFrame` 之间可见。

### 7.2 效果级关键帧

效果属性的动画通过**关键帧字符串**编码（见 [4.4 节](#44-关键帧字符串混合类型)），支持多种插值方式。几乎所有可动画化的属性（X/Y/Z 位置、旋转、缩放、透明度、颜色参数等）都可以包含关键帧数据。

## 8. 场景组织

### 8.1 场景与对象的关系

对象通过 `scene` 属性关联到场景。该属性**可能未显式声明**——此时对象的场景归属由其在原文件中的出现位置推断（出现在 `[scene.X]` 之后即属于该场景）。

```
complex.aup2: 247 个对象中，111 个有显式 scene= 属性，136 个靠位置推断
simple.aup2:  94 个对象中，22 个有显式 scene= 属性，72 个靠位置推断
```

### 8.2 区块序号 vs 内部 ID：两条独立轴

每个场景同时拥有两个标识符，各自承担不同的角色：

| | 区块序号 `[scene.N]` | 内部 ID `scene=` |
|------|------|------|
| **性质** | 文件中的排列位置 | 场景的永久标识 |
| **决定因素** | AviUtl UI 中标签页的拖拽顺序 | 场景创建时分配，之后不变 |
| **可变性** | 用户重排标签页时改变 | 不可变 |
| **谁引用它** | 无需被引用 | 对象通过 `scene=` 属性引用 |

`complex.aup2` 的完整数据清晰地展示了两者的分异（17 个场景中有 13 个 mismatch）：

```
区块序号 [scene.N]    内部 ID    名称        说明
─────────────────────────────────────────────────────
[scene.0]           scene=0     Root
[scene.1]           scene=1     Scene1
[scene.2]           scene=3     Scene3      ← Scene2 和 Scene3 互换了位置
[scene.3]           scene=2     Scene2
[scene.4]           scene=4     Scene4
[scene.5]           scene=6     Scene6      ← Scene6~11 集体上移一格
[scene.6]           scene=7     Scene7        （因为 Scene5 被拖到了后面）
[scene.7]           scene=8     Scene8
[scene.8]           scene=9     Scene9
[scene.9]           scene=10    Scene10
[scene.10]          scene=11    Scene11
[scene.11]          scene=5     Scene5      ← Scene5 从第 5 位拖到第 11 位
[scene.12]          scene=12    Scene12
[scene.13]          scene=14    Scene14     ← 内部 ID 13 缺失
[scene.14]          scene=15    ボツ
[scene.15]          scene=16    Scene16
[scene.16]          scene=17    Scene17
```

#### 设计动机

这是「**可变顺序 vs 不可变身份**」的经典解耦模式，有三个明确动机：

**1. 非破坏性重排**

用户在 AviUtl 中拖拽场景标签页时，只需移动 `[scene.N]` 区块在文件中的位置。如果区块序号和内部 ID 是同一个东西，每次重排就必须改写所有引用该场景的对象的 `scene=` 属性——在 247 个对象的项目中，这将是一场大规模、高风险的文件改写操作。

**2. 稳定的对象引用**

对象通过 `scene=5` 引用场景。无论 Scene5 被拖到标签栏的第几个位置，所有引用了它的对象不受影响。`scene=` 像数据库外键一样，指向的是身份而非位置。

**3. 删除不回收 ID**

内部 ID 13 缺失（从 12 直接跳到 14），说明 Scene13 曾被创建然后删除。AviUtl 没有回收这个 ID——永久空洞。这与数据库主键自增不回收的策略一致：**ID 是墓碑，不是序号**。避免已删除场景的残留引用意外复活到一个新创建的场景上。

#### 类比

```
区块序号 [scene.N]  =  UI 标签页位置（表现层，可变）
内部 ID  scene=     =  数据库主键（持久层，不变）

N 决定了文件中的排列 → 决定了 AviUtl 打开时的标签顺序
scene= 决定了对象的归属 → 重排标签页不影响任何对象关系
```

## 10. 总结

AUP2 是一种**扁平嵌套**的文本格式：

- **扁平**：所有区块在语法层面是平级排列的 `[X]` 标记
- **嵌套**：语义层面通过区块类型和出现顺序建立层级关系

核心设计模式：
1. **区块类型编码语义**：`[scene.X]` vs `[Y]` vs `[Y.Z]` 的类型决定了它们在层次结构中的位置
2. **顺序即结构**：对象的场景归属、效果的叠加次序都由在原文件中的出现顺序决定
3. **自定义关键帧编码**：动画数据使用 `逗号+管道符` 的紧凑格式内嵌在字符串值中
4. **ID 与序号分离**：场景的内部 ID 与区块序号可以不同，支持 UI 层重排序
